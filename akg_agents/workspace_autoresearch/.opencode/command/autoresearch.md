# AutoResearch (opencode) — Init / Resume / Run Optimization Loop

Single entry point: initialize a new task, resume an existing one, or kick
off the optimization. The autoresearch **plugin** drives every transition:
it blocks illegal actions and appends `[AR Phase: ...]` guidance to your
tool results — read that guidance after each tool call and never stop
between phases. (The plugin shares its decision logic with the Claude Code
build; behaviour is identical.)

## Arguments

`$ARGUMENTS` — one of:

- **`--resume`** or **`--resume <task_dir>`** — continue the most recent task
  (or the specified one).
- **Task dir** — resume that specific task: `ar_tasks/my_task_123456_abc`.
- **Init flags** — new task from a ref + seed kernel:
  ```
  --ref <file> --kernel <file> --op-name <name>
  --devices <N[,M,...]>
  [--max-rounds <N>] [--eval-timeout <sec>]
  [--no-code-checker] [--worker-url host:port[,...]]
  ```
  Both `--ref` and `--kernel` are required. `--kernel` is a seed `.py` file
  for single-file DSLs, or the project directory for directory-backed DSLs
  (`ascendc`, `ascendc_catlass`) whose sibling `kernel.py` is the entry.
  `--worker-url` routes eval to an already-running remote HTTP worker; if it
  is unreachable, stop with an actionable error — do not try to start or
  repair the worker from inside the loop.

`--output-dir` defaults to `ar_tasks`.

## Step 1 — Parse `$ARGUMENTS`

Run, as a single bash tool call:

```bash
python scripts/engine/parse_args.py $ARGUMENTS
```

It returns a one-line JSON dispatch record:
`{"mode": "scaffold|resume|ask", "command": "...", "argv": [...], "values": {...}, "missing": [...]}`.

`command` is the agent-facing Bash form. Headless drivers consume `argv`
directly without `shell=True`, so paths containing spaces or shell
metacharacters remain literal arguments.

**`values` is the single source of truth for every flag.** Do not modify,
substitute, or pull defaults from elsewhere.

## Step 2 — Dispatch by mode

- **`ask`** — show `missing` to the user and stop; re-invoke `/autoresearch`
  once the flags are complete. Don't guess.
- **`resume` / `scaffold`** — run `command` verbatim as a bash tool call,
  then read the resolved task_dir from the **last line of stdout**:
  - **scaffold** prints a JSON record `{"task_dir": "...", "status":
    "ok"|"error", ...}` — use its `task_dir` field, and stop if `status` is
    `"error"`.
  - **resume** prints the bare task_dir path.

  Non-zero exit → stop and report.

## Step 3 — Activate

```bash
export AR_TASK_DIR="<task_dir from step 2>"
```

Run this as its own bash tool call. The plugin sees it and activates the
task; the activation guidance (`[AR Phase: ...]`) is appended to that tool's
result. Follow it.

> Note: the activation persists through the task's own state file (keyed by
> your opencode session), not through the shell env — so it survives even if
> bash tool calls don't share a shell. You do not need to re-export it.

## Step 4 — Loop

Follow the phase guidance appended to each tool result. Never stop between
phases.

- **BASELINE** — `python scripts/engine/baseline.py "$AR_TASK_DIR"` (skipped
  automatically if scaffold already ran it).
- **PLAN / REPLAN** — two steps:
  1. Write `<items>...</items>` XML to `$AR_TASK_DIR/.ar_state/plan_items.xml`.
  2. `python scripts/engine/create_plan.py "$AR_TASK_DIR"` (no second arg).
  The plugin guidance carries the XML schema.
- **DIAGNOSE** — spawn the **`ar-diagnosis` subagent** (opencode subtask
  targeting the `ar-diagnosis` agent at `.opencode/agent/ar-diagnosis.md`).
  The neutral host guidance says "spawn the ar-diagnosis subagent"; in
  opencode that means the `task` tool with `agent: ar-diagnosis`. The
  subagent writes the diagnosis artifact to the path in the appended
  `[AR Phase: DIAGNOSE]` guidance; the host validates it, then you write
  `plan_items.xml` and run `create_plan.py`. After 5 failed attempts the gate
  relaxes to manual planning. Full contract: AGENTS.md invariant #10.
- **EDIT** — Edit only the files in `task.yaml`'s `editable_files`. When
  done: `python scripts/engine/pipeline.py "$AR_TASK_DIR"`. For Ascend
  profiling evidence, append `--trace`; the round keeps
  `kernel_details.csv`, `op_statistic.csv`, and `trace_view.json`.
  Later PLAN / REPLAN / DIAGNOSE guidance automatically points to them;
  read the CSVs first and the timeline JSON only when needed.
- **FINISH** — `.ar_state/report.md` is auto-generated by pipeline.py.
  Summarize and stop. Do not write any files.

## Rules

- Keep going between phases; the plugin blocks early stops.
- The plugin blocks wrong actions and tells you the next step in the tool
  result — read it.
- Never hand-edit `plan.md` or `.ar_state/state.json`; always go through the
  scripts.
- Never invent flag values not produced by `parse_args.py`.
