# 精度调试最佳实践

## 调试原则

### 1. 先怀疑公式，再怀疑实现

大部分精度问题是数学/算法本身的问题，而非实现错误。

**检查顺序**：
1. 数学公式是否正确
2. 公式到代码的映射是否正确
3. API 使用是否正确
4. 数据类型是否匹配
5. 数值稳定性问题

### 2. 从小样本开始

测试金字塔：
```
     ┌────────────┐
     │  完整测试   │  ← 最后验证
     ├────────────┤
     │  中等规模   │  ← 扩展验证
     ├────────────┤
     │  小规模     │  ← 基本验证
     ├────────────┤
     │  单元素     │  ← 最先测试
     └────────────┘
```

### 3. 记录一切

每次修改都要记录：
- 修改内容
- 误差变化
- 性能影响

```markdown
### 调试日志

| 修改 | Max Error | Mean Error | 性能 |
|-----|-----------|------------|------|
| 初始 | 1.2e-2 | 3.5e-4 | - |
| FP32 累加器 | 2.1e-4 | 1.2e-6 | -5% |
| 数值稳定算法 | 8.5e-6 | 2.3e-7 | -8% |
```

### 4. 理解才能解决

不要盲目试错，要理解误差来源：
- 为什么会有误差？
- 误差来自哪个步骤？
- 如何从根本上避免？

## 预防措施

### 设计阶段

1. **选择数值稳定的算法**
   - 避免 a-b 当 a≈b
   - 使用 log-sum-exp 避免溢出
   - 先归一化再计算

2. **规划混合精度方案**
   - 输入/输出 FP16
   - 关键中间计算 FP32
   - 累加类操作 FP32

3. **了解硬件约束**
   - 数据对齐要求
   - 最小元素数约束
   - 单次处理上限

### 编码阶段

1. **关键路径注释**
```cpp
// 精度敏感：使用 FP32 避免累加误差
float sum_fp32 = 0.0f;
for (int i = 0; i < size; ++i) {
    sum_fp32 += static_cast<float>(input[i]);
}

// 精度敏感：先减 max 避免 exp 溢出
half max_val = ReduceMax(input);
half shifted = input[i] - max_val;
```

2. **输入验证**
```cpp
// 检查硬件约束
if (cols < 8) {
    printf("Error: cols must be >= 8 (got %d)\n", cols);
    return;
}

// 检查数值范围
for (int i = 0; i < size; ++i) {
    if (isinf(static_cast<float>(input[i]))) {
        printf("Warning: input[%d] is Inf\n", i);
    }
}
```

3. **调试支持**
```cpp
#ifdef DEBUG_PRECISION
    printf("Step %d: value=%.6f\n", step, static_cast<float>(value));
#endif
```

## 容差设置指南

### 推荐容差

| 场景 | rtol | atol | 说明 |
|-----|------|------|------|
| **FP16 标准** | 1e-3 | 1e-4 | 浮点精度有限 |
| **FP32 标准** | 1e-5 | 1e-6 | 标准精度 |
| **整数** | - | 0 | 必须精确匹配 |

### 特殊场景容差

| 场景 | rtol | atol | 原因 |
|-----|------|------|------|
| Softmax FP16 | 1e-3 | 1e-4 | 概率输出，精度有限 |
| Softmax FP32 | 1e-5 | 1e-6 | 标准精度 |
| Reduce FP16 | 5e-3 | 1e-4 | 累加误差较大 |
| Reduce FP32 | 1e-5 | 1e-6 | 标准精度 |
| exp/log FP16 | 1e-3 | 1e-4 | 超越函数精度有限 |
| 三角函数 FP16 | 5e-3 | 1e-4 | 泰勒展开近似 |

### 容差选择原则

1. **根据应用场景**
   - 科学计算：严格要求
   - 深度学习：相对宽松
   - 可视化：最宽松

2. **根据数据类型**
   - FP16：宽松容差
   - FP32：标准容差
   - INT：零容差

3. **根据算子特性**
   - 逐元素操作：标准容差
   - Reduce 操作：宽松容差
   - 超越函数：宽松容差

## 常见错误模式

### 错误 1：过度调试

**症状**：反复尝试不同方法，没有系统性

**正确做法**：
1. 先分析误差模式
2. 识别可能根因
3. 针对性验证
4. 记录每次尝试

### 错误 2：忽略 FP32

**症状**：直接用 FP16 调试，难以区分精度问题

**正确做法**：
1. 先用 FP32 验证算法正确性
2. 再测试 FP16 精度
3. 对比差异定位问题

### 错误 3：盲目提高精度

**症状**：所有计算都用 FP32，性能下降

**正确做法**：
1. 识别真正需要高精度的步骤
2. 只在关键步骤使用 FP32
3. 平衡精度和性能

### 错误 4：忽略硬件约束

**症状**：特定输入规模结果异常

**正确做法**：
1. 查阅硬件约束文档
2. 添加输入验证
3. 在文档中说明限制

## 调试效率技巧

### 1. 诊断先行

```
问题 → 误差分析 → 模式识别 → 根因推断 → 针对性修复
```

### 2. 快速验证

```
单元素 → 小规模 → 标准规模 → 大规模 → 边界值
```

### 3. 计数规则

```
快速方法计数 < 7 → 继续
快速方法计数 >= 7 → 切换二分调试
```

## 性能与精度平衡

### 混合精度设计

| 部件 | 精度选择 | 原因 |
|-----|---------|------|
| 输入 | FP16 | 节省带宽 |
| 输出 | FP16 | 节省存储 |
| 中间计算 | FP32 | 提升精度 |
| 累加器 | FP32 | 避免累积误差 |
| 比较 | FP16 足够 | 不影响精度 |

### 性能评估

修复精度问题后评估性能影响：
- 性能损失 < 5%：可接受
- 性能损失 5-10%：需评估
- 性能损失 > 10%：考虑优化方案

## 文档化

### 算子文档应包含

1. **精度说明**
   - 支持的数据类型
   - 各数据类型的精度特性
   - 推荐容差设置

2. **已知限制**
   - 硬件约束（对齐、最小元素数）
   - 精度限制（FP16 有效数字）
   - 数值范围限制

3. **使用建议**
   - 推荐的数据类型组合
   - 需要避免的输入模式
   - 性能优化提示
